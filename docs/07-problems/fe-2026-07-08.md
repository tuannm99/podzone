## P0: Correctness Bugs

### FE1 — `useOrderActions.ts:55` — `setSubmitting(true)` without try/finally

`createMockOrder` sets `setSubmitting(true)` at line 55, then `await`s
`createRoutedOrder`. If the transport layer throws (network error, timeout),
`setSubmitting(false)` at line 64 is never reached. The "Create Order" button
stays permanently disabled for the session.

Fix: wrap lines 55–76 in `try { ... } finally { params.orderForm.setSubmitting(false) }`.

(tracked — "Form Submission Is Inconsistent" lists this path; concrete location added here)

### FE2 — `TenantOrdersPageView.tsx:268-277` — `loadOrders`/`loadCandidates` fire-and-forget with no stale guard

Two `void` calls inside a `createEffect` have no version counter:

```ts
createEffect(() => {
    ...
    void loadCandidates()
    void loadOrders()
})
```

`loadQueuePage` in the same file (line 297) is properly guarded by a
`queueRequestVersion` counter. `loadOrders` and `loadCandidates` are not. When
the user switches tenant or store while either fetch is in-flight, a slow
response from the old context can call `setOrders(oldTenantOrders)` after the
new context has already loaded, silently displaying wrong-tenant data.

Fix: add a request-version counter to `loadOrders` and `loadCandidates`, matching the
`loadQueuePage` pattern already in the same file.

(tracked — effect-driven fetching listed; stale-guard gap for these two calls is new)

### FE3 — `createBootstrapLoader.ts:9` — `setLoading(true)` without try/finally

```ts
state.setLoading(true)
const tenantResult = await listUserTenants(userID)
if (currentRequest !== requestID) return   // stale-request guard, ok
...
state.setLoading(false)
```

If `listUserTenants` throws (network error, 5xx not caught in the service
adapter), `setLoading(false)` is never reached. The IAM console becomes
permanently stuck in the loading state with no recovery affordance and no
error message shown.

Fix: wrap the await in try/finally, or ensure `listUserTenants` never throws by
returning an error result for all transport failures.

## P1: Stale Data And Double-Submit

### FE4 — `TenantOrdersPageView.tsx:317-323` — `loadRoutingRecommendation` has no stale guard

`loadRoutingRecommendation` is called via `void` from a `createEffect` that
tracks `orderForm.values` and `workspaceReady`. Rapid product/region changes
fire concurrent requests; the slowest (stale) response can overwrite the
current recommendation. No version counter or abort controller is present inside
the loader.

Fix: same version-counter approach as `loadQueuePage`.

### FE5 — `createPlatformRolesViewModel.ts:75` / `createTeamAccessViewModel.ts` — fire-and-forget remove with no saving state

`remove` in `createPlatformRolesViewModel` awaits `removePlatformRole` without
any `setSaving` guard. The remove button in `PlatformRolesTable.tsx` is not
disabled during the request. A fast double-click fires two concurrent remove
requests for the same role/member; the second request fails with a
not-found or duplicate error that surfaces as an unhandled rejection.

Same pattern in `createTeamAccessViewModel.ts`.

Fix: add `setSaving(true)` / `finally { setSaving(false) }` and expose `saving`
to disable the button while the request is in-flight.

### FE6 — `createPipelineViewModel.ts:42-46` — auto-select effect resets explicit user selection on every reload

```ts
createEffect(() => {
    const items = requests.items()
    if (items.some((request) => request.id === selectedRequestId())) return
    setSelectedRequestId(items[0]?.id || '')
})
```

After `requests.reload()` (e.g., after a retry), `items()` briefly returns the
stale collection or an empty array while loading. During that window the
`some(...)` check fails and the selection resets to `items[0]?.id || ''`. When
the reload completes, the effect re-runs and picks `items[0]` again — not the
request the user had explicitly selected. The user's selection is lost on every
mutation that reloads the list.

Fix: skip the auto-select effect when the resource is in a loading state
(`requests.loading()` is true), preserving the current selection until the
new data arrives.

## P1: Architecture

### FE7 — `AdminHomePage.tsx:140` — `window.location.href` assignment for internal SPA navigation

```ts
window.location.href = `/t/${normalizedTenantID}?storeId=${encodeURIComponent(normalizedStoreID)}`
```

Assigning `window.location.href` triggers a full document reload on every store
switch. All reactive state, resource caches, and Solid signal trees are torn
down and rebuilt from scratch, including any other tabs the user had open. A
router `navigate()` call is the correct primitive here.

Fix: `import { useNavigate } from '@solidjs/router'` and call
`navigate(\`/t/${normalizedTenantID}?storeId=${encodeURIComponent(normalizedStoreID)}\`)`.

(tracked — hash/history patterns listed; `window.location.href` hard-assign is a new location)

## P2: Reactivity And Consistency

### FE8 — `createClientPagination.ts:11-23` — reference-equality check resets page on every refetch

```ts
let previousItems = items()

createEffect(() => {
    const currentItems = items()
    if (currentItems !== previousItems) {
        previousItems = currentItems
        setPage(1)
        return
    }
    ...
})
```

Every refetch (resource reload, signal update) produces a new array object.
`currentItems !== previousItems` is always true on a refetch even if the
items are unchanged in content, so the page resets to 1 on every background
poll, losing the user's position mid-navigation.

Fix: replace reference-equality with a length + first-element-id check, or keep
a stable `items` memo from the server and only reset page when the total count
or first-page fingerprint changes. Alternatively, expose `keepPage: boolean` to
callers for controlled resets.

### FE9 — `RegisterPage.tsx:10-15` — per-field signals instead of a typed form store

Four separate `createSignal` calls for `username`, `email`, `password`, and
`confirmPassword`. The style guide requires one cohesive typed `createStore` per
form. Without a form store there is no field-level validation state, no
touched/dirty tracking, and no structured server-error mapping. The manual
`password() !== confirmPassword()` check in the submit handler is ad-hoc and
not reusable.

Fix: replace the four signals with a `createStore<RegisterFormValues>` following
the same pattern as `createInvitesViewModel` or any other onboarding form.

### FE10 — `OrdersInsightsPanel.tsx:43,64,110` — computed arrays silently truncated to 6

Three derived arrays (`forcedRerouteSummary`, `reconciliationOrders`,
`partnerFinanceSummary`) are sliced to 6 items with no count, "show more"
affordance, or indicator that items were omitted:

```ts
forcedRerouteSummary().slice(0, 6)
reconciliationOrders().slice(0, 6)
partnerFinanceSummary().slice(0, 6)
```

A store with 20 partners needing finance attention silently shows 6. The user
has no way to reach the rest. This is a pure client-side slice on data already
in memory, not server pagination.

Fix: either show a count badge ("6 of 20") with a link to the full collection
view, or remove the slice and let the panel scroll/paginate to all items.

## Shared Component Bugs

### FE11 — `Primitives.tsx:150` — `FieldLabel` breaks label-input association for all 9 field components

`FieldLabel` renders `<label>` without `for=` or `id` threading. No field
component generates or passes an `id` to its native control. For
`RadioGroupField`, a wrapping `<label>` over multiple radio inputs is
semantically wrong — it should be `<fieldset>`/`<legend>`. Click-to-focus,
screen reader announcement, and form autofill may all fail.

Fix: each field primitive must generate `id = props.id ?? createUniqueId()`,
set `for={id}` on the label and `id={id}` on the control, and add
`aria-invalid` / `aria-describedby` when an error is present. See
`docs/03-architecture-detail-design/design-system.md` → Component API Conventions.

### FE12 — `IamKeyValueBuilder.tsx:50` — `createEffect` overwrites in-progress edits on every parent re-render

```ts
const [entries, setEntries] = createSignal(normalizeEntries(props.value))
createEffect(() => {
    setEntries(normalizeEntries(props.value))  // fires on every props.value change
})
```

`commit()` calls `props.onChange(serialized)` → parent updates the reactive
signal → effect fires again and resets `entries` to the just-serialized value.
On a slow parent, this replaces a partially-typed key mid-keystroke. Also runs
twice on mount (signal init + first effect run).

Fix: remove the `createEffect`; initialize once from `props.value`. For
external reset, use an explicit `reset()` prop or a version counter to
distinguish external vs. internal changes.

### FE13 — `Overlay.tsx:~158` — Modal and Drawer have no focus trap or ARIA dialog semantics

`Modal` and `Drawer` render fixed-position containers without `role="dialog"`,
`aria-modal="true"`, `aria-labelledby`, focus trap, auto-focus on open, or
focus restore on close. Keyboard tab can escape the overlay into obscured
background content. Screen readers do not announce the overlay as a dialog.

Fix: see `docs/03-architecture-detail-design/design-system.md` → Gap G2 for the `useFocusTrap`
implementation.

### FE14 — `Primitives.tsx:8` — `ButtonColor` 'blue' and 'dark' are identical and render black

Both `'blue'` and `'dark'` map to `bg-gray-950 text-white` (identical class
strings). The default color is called `'blue'` but renders near-black. Callers
passing `color="blue"` expecting a blue button get a black one.

Fix: rename to `ButtonVariant` with values `'primary' | 'secondary' | 'danger' |
'success' | 'ghost'`; back with CSS custom properties. Run a codemod for
existing `color="blue"` → `color="primary"`.
