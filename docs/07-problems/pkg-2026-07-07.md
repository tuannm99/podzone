## K1 — `KVStore` interface has no `context` parameter; all operations ignore caller cancellation

**Resolved as of 2026-07-11.** `pkg/toolkit/kvstores/store.go` now has
`ctx context.Context` as the first parameter on `Get`, `GetKVs`, `Put`, `Del`.

**File:** `pkg/toolkit/kvstores/mongo.go:53,68,94,114`
**Severity:** High — can hang callers past request/shutdown deadlines

The `KVStore` interface defines `Get`, `GetKVs`, `Put`, `Del` without a `context.Context`
parameter. `MongoStore` creates its own:

```go
ctx, cancel := context.WithTimeout(context.Background(), mongoOperationTimeout)
```

on every call. Consequences:
- An HTTP handler cancelled at its deadline still waits up to 5 s for the MongoDB round-trip.
- A shutting-down service waits for all in-flight KV calls even after the root context is done.
- `KVPlacementResolver.Resolve` and `KVClusterRegistry.GetCluster` accept a `ctx` argument
  but can't forward it; callers get a false sense of propagation.

**Fix:** Add `ctx context.Context` as the first parameter to every `KVStore` interface method
and thread it through `MongoStore` (replace `context.Background()` with the passed-in `ctx`,
keep the 5 s as a min-timeout floor via `context.WithDeadline` capped at 5 s if desired).

## K2 — `CloseIdleDedicated` not on `Manager` interface and not wired to any lifecycle timer

**File:** `pkg/pdtenantdb/manager.go:245`, `pkg/pdtenantdb/fxmodule.go`
**Severity:** High — dedicated database pools accumulate unbounded until shutdown

`Manager` interface exposes only `DBForTenant`, `WithTenantTx`, `CloseAll`. The method:

```go
func (m *managerImpl) CloseIdleDedicated(now time.Time) { ... }
```

is inaccessible through the interface. `fxmodule.go` wires only `CloseAll` to the `OnStop`
hook. No periodic goroutine or worker calls `CloseIdleDedicated`. Result: dedicated pools
accumulate for every distinct tenant until shutdown (`MaxDedicatedPools=200` default, but
nothing enforces the cap during the run — `reserveDedicatedLocked` returns an error when the
cap is exceeded, but idle pools are never reaped during runtime).

**Fix:** Add `CloseIdleDedicated(now time.Time)` to the `Manager` interface. Wire a
background ticker (e.g. every `DedicatedIdleTTL/2`) in `fxmodule.go` using `fx.Lifecycle`
to call it periodically.

## K3 — `waitUntilReady` blocks the Sarama partition goroutine for the full retry delay

**File:** `pkg/messaging/kafka/consumer.go:131–143` (the `waitUntilReady` function)
**Severity:** High — head-of-line blocking; a single retry message stalls all subsequent messages in its partition

Sarama assigns one `ConsumeClaim` goroutine per partition. When a retry message arrives
with `NextAttemptAt` in the future:

```go
timer := time.NewTimer(delay)
select {
case <-ctx.Done(): return ctx.Err()
case <-timer.C:    return nil
}
```

The goroutine parks for `delay` (up to `MaxDelay`). All following messages in that partition
queue behind it. A retry topic that receives bursts (e.g. 100 messages with 30-minute delays)
blocks the partition for 30 minutes per message, effectively serializing a multi-hour backlog.

**Fix:** Use a separate scheduling mechanism — either a time-priority heap with a ticker, or
route delayed retry messages to a dedicated "scheduled" topic processed by a single-threaded
scheduler goroutine. At minimum, do not park the Sarama partition goroutine.

## K4 — Malformed envelope causes infinite Kafka consumer restart loop

**File:** `pkg/messaging/kafka/consumer.go:93–100` (`consumeMessage` returning `decodeEnvelope` error)
**Severity:** High — a single non-JSON Kafka message wedges the consumer indefinitely

If `decodeEnvelope` fails (bad JSON, wrong schema), `consumeMessage` returns the error →
`ConsumeClaim` returns the error → Sarama ends the consumer group session → the group
immediately re-joins → the same message is consumed again from the uncommitted offset.
This loops forever: the malformed message is never committed, never dead-lettered, and blocks
all subsequent messages in the partition.

**Fix:** In `consumeMessage`, treat decode failures as a drop or dead-letter, not a session
termination:

```go
env, err := h.decodeEnvelope(msg)
if err != nil {
    h.observe(ctx, msg, messaging.Envelope{}, messaging.FailureActionDrop, "decode_error", err)
    session.MarkMessage(msg, "")
    return nil  // or publish to DLQ
}
```

## K5 — `InboxStore.Begin`: `ErrNoRows` after INSERT conflict returns `InboxDecisionAcquired`

**File:** `pkg/messaging/sqlstore/inbox_store.go:82–83`
**Severity:** Medium — idempotency guard can be bypassed under a delete-between-steps race

The `Begin` flow:
1. UPDATE reset-failed → 0 rows
2. INSERT ON CONFLICT DO NOTHING → 0 rows (conflict: row existed)
3. SELECT status → `sql.ErrNoRows`

Step 3 returns `ErrNoRows` when the row that blocked the INSERT has been deleted between
step 2 and step 3. The code treats this as `InboxDecisionAcquired` ("go ahead and process").
If the deleted row was a completed record (e.g. cleaned up by a maintenance job), the same
message will be reprocessed without idempotency protection.

**Fix:** On `ErrNoRows`, retry the INSERT once (the blocking row is gone) rather than
immediately returning `Acquired`. If the second INSERT also conflicts, fall through to the
SELECT path normally.

## K6 — `Registry.Handle` silently returns `nil` for unregistered message types

**Resolved as of 2026-07-11, with a follow-on fix.** `Registry.Handle` now
returns `fmt.Errorf("%w for type %q", ErrHandlerNotFound, msg.Type)` per the
original suggested fix below — but applying that blindly at every call site
turned out to be too strict: `internal/auth/controller/eventhandler/iamprojection`
only handles a subset of event types on its topic, and returning a raw error
for the rest triggered `FailureActionReturn` in the Kafka consumer, which
does not ack the message (see `pkg/messaging/kafka/consumer.go:214`) —
effectively re-introducing a stuck/retry problem instead of a silent-drop
one. Fixed by wrapping the registry in `NewHandler` (`handler.go`) so
`errors.Is(err, messaging.ErrHandlerNotFound)` no-ops for that specific
consumer, while `Registry` itself keeps erroring by default (safe default
for callers that don't opt in). This is the "explicit `DropHandler`"
semantics the original fix note below anticipated.

**File:** `pkg/messaging/registry.go:36–38`
**Severity:** Medium — message routing bugs are invisible; messages are lost without error

```go
handler, ok := r.handlers[msg.Type]
if !ok {
    return nil   // silent drop
}
```

When a new message type is deployed without a registered handler (or when a handler is
mis-registered with the wrong type string), `Handle` returns `nil`. The idempotency
middleware and outbox relay interpret `nil` as "handled OK" and mark the message as
complete. The message is permanently lost with no log, metric, or error.

**Fix:** Return an error for unregistered types:

```go
if !ok {
    return fmt.Errorf("messaging: no handler registered for type %q", msg.Type)
}
```

If intentional "ignore" semantics are needed, require callers to register an explicit
`DropHandler` for those types.

## K7 — `IdempotentConsumer` silently bypasses deduplication when `msg.ID == ""`

**File:** `pkg/messaging/idempotency.go:23`
**Severity:** Medium — at-least-once guarantee degrades silently

```go
if store == nil || strings.TrimSpace(msg.ID) == "" {
    return next.Handle(ctx, msg)
}
```

A publisher that omits `Envelope.ID` skips the inbox entirely. No log, no metric. Combined
with K6 (unregistered type → silent drop), an untracked message can be both processed and
lost with no observable signal.

**Fix:** Treat missing `msg.ID` as an error, or derive a synthetic ID from
`(consumerName, topic, partition, offset)` before entering idempotency check. At minimum
emit an observation event so silent bypasses are visible.

## K8 — Relay calls `MarkPublished` per-message; mid-batch failure leaves published messages unmarked

**File:** `pkg/messaging/kafka/relay.go:41-47`
**Severity:** Medium — N round-trips + republish duplicates on mark failure

```go
for _, item := range items {
    if err := r.publisher.Publish(ctx, ...); err != nil { ... continue }
    if err := r.store.MarkPublished(ctx, []string{item.ID}, now); err != nil {
        return err  // exits loop; previous items already Kafka-published, not marked
    }
}
```

`MarkPublished` accepts `[]string` (designed for batching), but the call site passes a
single element. If `MarkPublished` fails midway, all preceding items are Kafka-published
but still `pending` — they are re-sent on the next relay tick, producing duplicates.

**Fix:** Collect successfully published IDs, then call `MarkPublished` once after the loop.

## K9 — `os.ExpandEnv` on raw YAML bytes silently corrupts secrets containing `$`

**File:** `pkg/pdconfig/config.go:29`
**Severity:** Medium — silent credential corruption in production

```go
expanded := os.ExpandEnv(string(b))
```

Any `$NAME` or `${NAME}` sequence in the YAML is substituted from the process environment
before the YAML parser runs. Database passwords or API tokens containing `$` (e.g.
`p@$$word`) are silently truncated or replaced with another env var's value, producing an
incorrect credential with no error.

**Fix:** Restrict expansion to explicitly marked placeholders (e.g. `!env TAG` in YAML), or
use `os.Expand` with a whitelist function instead of `os.ExpandEnv` on the whole file.

## K10 — CORS `AllowOriginFunc` uses developer's username as an allowed-origin substring

**File:** `pkg/pdgrpcgateway/server.go:21`
**Severity:** High — CORS bypass; any origin containing "tuannm" or "podzone" passes

```go
if strings.Contains(o, "tuannm") || strings.Contains(o, "podzone") {
    return true
}
```

`http://evil-tuannm.attacker.com` satisfies the first check. `http://podzone-phishing.com`
satisfies the second. Both bypass CORS, enabling cross-site requests from arbitrary origins
to the gRPC-gateway with the user's credentials.

**Fix:** Allowlist exact origins or use a suffix anchor:

```go
allowedSuffixes := []string{".podzone.io", ".podzone.dev"}
for _, suffix := range allowedSuffixes {
    if strings.HasSuffix(o, suffix) { return true }
}
```

Keep the existing `localhost`/`127.0.0.1` prefix check for local dev — that part is correct.
