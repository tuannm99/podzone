## P0: Correctness Bugs

### FE15 ‚Äî `useOrderStorage.ts:87` ‚Äî `applyQueuePreset` silently drops queueView and queueSort (CONFIRMED)

`applyQueuePreset` calls `setActiveQueueView`, `setActiveQueueSort`, `setOperatorLens` in
sequence (lines 87‚Äì89). Each setter now does:
```ts
void navigate({ search: { ...search(), key: val } })
```
TanStack Router's `navigate` is async (commits via microtask). All three spreads read
the same pre-call `search()` snapshot. The last spread (`operatorLens`) overwrites the
previous two ‚Äî `queueView` and `queueSort` from the preset are silently dropped.

**Fix:** Combine all three into a single `navigate` call:
```ts
void navigate({
    search: {
        ...search(),
        queueView: preset.queueView,
        queueSort: preset.queueSort,
        operatorLens: preset.operatorLens,
    },
})
```

**Files:** `src/modules/backoffice/pages/orders/useOrderStorage.ts:87`

### FE16 ‚Äî `useOrderActions.ts:83` ‚Äî 8 action functions have no in-flight guard (CONFIRMED)

`advanceOrder`, `raiseException`, `saveShipment`, `saveSettlement`, `saveIssueHandling`,
`saveQueueControl`, `rerouteBlockedOrder`, `applyBulkUpdate` ‚Äî none have `isSaving` flag
or early-return guard. Only `createMockOrder` (line 55) has `orderForm.setSubmitting`.

Double-clicking any action button fires two concurrent mutations. Both may succeed; both
call `setOrders` mapping the same orderId; second response silently overwrites the first;
`onChanged` reloads twice in parallel.

**Fix:** Add `const [isSaving, setIsSaving] = createSignal(false)` pattern (as done for
`createMockOrder`). Gate each action: `if (isSaving()) return; setIsSaving(true)`.
Expose `isSaving` to disable buttons while in-flight.

**File:** `src/modules/backoffice/pages/orders/useOrderActions.ts`

### FE17 ‚Äî `createOrderAuditViewModel.ts:121` ‚Äî feedResource + loadEntries race ‚Üí mixed-filter data (CONFIRMED)

`feedResource`'s `createEffect` (line 121‚Äì141) calls `setEntries(result.data.entries)` ‚Äî
a full reset ‚Äî independently of `loadEntries`'s `requestVersion` counter. A filter change
while `loadEntries(append=true)` is in-flight:

1. `feedResource` resets `entries` to new-filter data
2. Old `loadEntries` resolves ‚Äî `requestVersion` still matches (not incremented by filter
   change) ‚Üí appends old-filter rows to the already-reset new-filter list

**Fix:** Increment `requestVersion` (or a separate `loadMoreVersion`) whenever `applyFilters`
is called. In `loadEntries`, check the version after each `await`. Alternatively, cancel
in-flight `loadEntries` when a filter change fires.

**File:** `src/modules/backoffice/pages/order-audit/createOrderAuditViewModel.ts`

### FE18 ‚Äî `createTenantOrdersViewModel.ts:342` ‚Äî `setOperatorLens` navigate inside `createEffect` (CONFIRMED)

A `createEffect` on `ordersResource.latest` calls `setOperatorLens(firstAssigned.operatorAssignee)`
when `operatorLens()` is empty. Since `setOperatorLens` now does `navigate(...)`, every
orders reload (after any mutation) pushes a history entry and retriggers `queuePageResource`
(whose source key includes `operatorLens()`), causing a second API fetch on every order load.

**Fix:** This auto-lens logic predates the URL-state migration. Either:
- Keep `operatorLens` as a local `createSignal` (not URL-backed) since it is auto-populated
  from data, not a user-navigable filter
- Or replace `setOperatorLens(...)` with a direct `navigate({ replace: true, search: ... })`
  to avoid polluting history

**File:** `src/modules/backoffice/pages/orders/createTenantOrdersViewModel.ts:342`

## P1: Stale Data / UX Bugs

### FE19 ‚Äî `createOrderAuditViewModel.ts:24` ‚Äî Applied filters not reset on workspace switch (CONFIRMED)

`appliedActorFilter`, `appliedOrderFilter`, `appliedPartnerFilter`, `appliedAssigneeFilter`
are `createSignal('')` and are never reset in any `createEffect` tracking
`options.tenantID()` or `options.storeID()`. On workspace switch, `feedResource` refetches
for the new context but still includes stale applied filter values ‚Äî the narrowed feed is
shown with no UI affordance that filters are active.

**Fix:** Add a `createEffect` on `options.tenantID` / `options.storeID` that resets all
four applied filter signals (and the raw filter signals) to `''`.

**File:** `src/modules/backoffice/pages/order-audit/createOrderAuditViewModel.ts`

### FE20 ‚Äî `QueueToolbarPanel.tsx:15` ‚Äî `setOperatorLens` fires navigate on every keypress (CONFIRMED)

Line 15: `onInput={(event) => board.setOperatorLens(event.currentTarget.value)}` ‚Äî no
debounce, no apply button. `setOperatorLens` is a URL navigate setter. Each character
pushes a history entry and fires a `queuePageResource` refetch.

Contrast: `queueSearch` (line 22) has an explicit "Search" button guarded by `applyQueueSearch`.

**Fix:** Keep `operatorLens` input bound to a local signal (like `queueSearch`); add an
"Apply" step that commits to URL. Or debounce the navigate call (min 300ms).

**File:** `src/modules/backoffice/pages/orders/QueueToolbarPanel.tsx:15`

### FE22 ‚Äî `createTenantOrdersViewModel.ts:178` ‚Äî `|` in user search collides with pipe-join key (CONFIRMED)

The `queuePageResource` source key:
```ts
[tenantId, storeId, page, appliedQueueSearch, activeQueueView, activeQueueSort, operatorLens.trim()].join('|')
```
If `appliedQueueSearch` contains `|` (e.g. `"foo|all"`), the joined string becomes
ambiguous with a different combination of field values ‚Äî `createResource` considers the
source unchanged and skips the refetch, showing stale results.

**Fix:** Use `JSON.stringify(...)` or encode each segment: `.map(encodeURIComponent).join('|')`.
Or use `JSON.stringify([tenantId, storeId, page, ...])`.

**File:** `src/modules/backoffice/pages/orders/createTenantOrdersViewModel.ts:178`

## P2: Architecture / Conventions

### FE23 ‚Äî `TenantHomeSections.tsx:1` ‚Äî Panel imports `tokenStorage` directly (CONFIRMED)

Line 1: `import { tokenStorage } from '@/services/tokenStorage'` ‚Äî panel reads a storage
adapter directly, bypassing the ViewModel layer.

**Rule:** `agent/SOLID_STYLE_GUIDE.md` ‚Äî "Panels render and invoke feature actions; they
never import from `@/services/` or read storage adapters directly."

**Fix:** Move the `tokenStorage.getActiveTenantID()` read into the parent ViewModel and
pass the value down as a prop or via context.

**File:** `src/modules/backoffice/pages/home/TenantHomeSections.tsx:1`

---

### FE24 ‚Äî `createTenantOrdersViewModel.ts:270` ‚Äî `insightsContextValue.tenantId` non-reactive snapshot (PLAUSIBLE)

`params().tenantId` is evaluated once at ViewModel construction and stored as a plain string
in `insightsContextValue`. If the ViewModel is ever provided above the route boundary or
kept alive across tenant navigation (e.g., via a persistent context provider), `tenantId`
in the insights context goes stale and insight API calls send requests for the wrong tenant.

**Fix:** Store `() => params().tenantId` (a reactive accessor) instead of the snapshot.

**File:** `src/modules/backoffice/pages/orders/createTenantOrdersViewModel.ts:270`
