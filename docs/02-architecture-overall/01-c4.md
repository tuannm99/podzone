# C4 Architecture Index

This file is the C4 entrypoint for Podzone.

## C1: System Context

Read:

- [`02-system-context.md`](./02-system-context.md)

Purpose:

- identify users, external systems, and Podzone as a system.

Use before:

- changing actors;
- adding external integrations;
- changing product boundary.

## C2: Container View

Read:

- [`03-containers.md`](./03-containers.md)

Purpose:

- identify deployable services, databases, queues, gateways, and frontend
  containers.

Use before:

- adding a service;
- changing service communication;
- changing deployment topology.

## C3: Component/Module View

Read:

- [`04-data-ownership.md`](./04-data-ownership.md) — system-wide data ownership map
- [`03-architecture-detail-design/01-modules.md`](../03-architecture-detail-design/01-modules.md)
- [`03-architecture-detail-design/02-bounded-contexts.md`](../03-architecture-detail-design/02-bounded-contexts.md)
- Per-service component view:
  [`03-architecture-detail-design/services/<name>/README.md`](../03-architecture-detail-design/services/README.md)
  ("C3: Component View" section in each service's `api-design.md`)

Purpose:

- identify modules inside services and data ownership boundaries.

Use before:

- refactoring service internals;
- applying DDD;
- changing repository ownership;
- adding cross-service calls.

## C4: Code/Sequence Details

Read:

- [`03-architecture-detail-design/04-code-map.md`](../03-architecture-detail-design/04-code-map.md)
- [`03-architecture-detail-design/05-transport-contracts.md`](../03-architecture-detail-design/05-transport-contracts.md)
- [`03-architecture-detail-design/07-async-messaging.md`](../03-architecture-detail-design/07-async-messaging.md)
- Per-service sequence diagrams (one per usecase):
  [`03-architecture-detail-design/services/<name>/api-design.md`](../03-architecture-detail-design/services/README.md)
  ("C4: Sequences Per Usecase" section in each service's `api-design.md`) —
  a sequence diagram lives next to the API/usecase it describes, not in a
  standalone cross-cutting file.

Purpose:

- explain code packages, important flows, transport contracts, and async event
  flow.

Use before:

- implementing a vertical slice;
- changing generated contracts;
- changing worker/outbox/event behavior.

## Required Traceability

Every architecture-significant sprint task should link:

```text
SRS requirement
  -> C4 doc
  -> contract or DB note
  -> sprint task
  -> verification result
```
